# План внедрения: съемка с камеры и запуск текущего пайплайна

## 1) Цель и критерий готовности
- Цель: добавить в UI кнопку **«Сделать снимок»**, которая открывает экран камеры, позволяет пользователю управлять съемкой (зум и вспышка), делает фото и сразу запускает существующий анализ на полученном кадре.
- Definition of Done:
  - на основном экране есть третья кнопка `Сделать снимок`;
  - по нажатию открывается встроенный экран камеры внутри приложения (не внешний интент-камера);
  - на экране камеры доступны: предпросмотр, съемка, управление вспышкой, зум (пинч/слайдер);
  - после успешного кадра экран камеры закрывается, выполняется `_runAnalysis(...)` для нового изображения;
  - ошибки камеры/разрешений показываются в понятном виде и не приводят к падению;
  - существующие сценарии «Выбрать из галереи» и «Выбрать файл» не ломаются.

## 2) Текущее состояние (по коду)
- `lib/main.dart`: есть 2 входа в пайплайн (`_pickFromGallery`, `_pickFromFiles`) и единый `_runAnalysis(String inputUri)`.
- Пайплайн уже принимает URI/путь файла и работает с изображениями/ PDF на Android через `MethodChannel('seed_detect/analyzer')`.
- В `pubspec.yaml` есть `image_picker`, но нет `camera` (нет встроенного экрана с контролями).
- `android/app/src/main/AndroidManifest.xml`: отсутствует `android.permission.CAMERA`.

## 3) Архитектурное решение
- Использовать пакет `camera` для встроенного экрана камеры и контролей (flash/zoom), т.к. `image_picker` с `ImageSource.camera` делегирует съемку внешнему приложению камеры и не гарантирует нужный UX.
- Текущий пайплайн анализа не менять: после съемки передавать путь кадра в уже существующий `_runAnalysis` через `Uri.file(photo.path).toString()`.
- Экран камеры выделить в отдельный виджет/файл для изоляции платформенной логики и чтобы не раздувать `main.dart`.

## 4) Пошаговый план реализации

### Шаг 0. Подготовка зависимостей и разрешений
Файлы: `pubspec.yaml`, `android/app/src/main/AndroidManifest.xml`

1. Добавить зависимость `camera` (при необходимости `camera_android_camerax` как backend для Android).
2. В манифест добавить `android.permission.CAMERA`.
3. Добавить `<uses-feature android:name="android.hardware.camera.any" android:required="false" />`, чтобы приложение могло устанавливаться и на устройства без камеры.

Результат: проект готов к инициализации встроенной камеры.

### Шаг 1. Реализовать экран съемки
Файлы: новый файл, например `lib/camera_capture_screen.dart` (или `lib/features/camera/camera_capture_screen.dart`)

1. Получать список камер через `availableCameras()` и выбирать основную заднюю (fallback на первую доступную).
2. Создать `CameraController` с разумным пресетом (`ResolutionPreset.high`/`medium` по балансу качество-производительность).
3. Показать `CameraPreview` + overlay-контролы:
   - кнопка спуска;
   - переключатель вспышки (`off/auto/always/torch` по поддержке);
   - управление зумом (пинч +/или слайдер в пределах `minZoom`..`maxZoom`).
4. При `takePicture()` возвращать путь кадра через `Navigator.pop(context, XFile)`.
5. На ошибки `CameraException`/недоступность камеры показывать user-facing сообщение и кнопку закрытия.
6. Корректно освобождать контроллер в `dispose()`.

Результат: независимый экран камеры, возвращающий готовый кадр в вызывающий экран.

### Шаг 2. Интеграция с текущим экраном анализа
Файл: `lib/main.dart`

1. Добавить кнопку `Сделать снимок` рядом с существующими кнопками выбора.
2. Реализовать метод `_captureFromCamera()`:
   - перевести экран в `AnalyzeState.picking`;
   - открыть `CameraCaptureScreen` через `Navigator.push`;
   - при отмене вернуть `AnalyzeState.idle`;
   - при успешном фото сформировать `inputUri` и вызвать `_runAnalysis(inputUri)`.
3. Оставить единый путь обработки ошибок через текущие `_state/_errorMessage`.
4. Заблокировать кнопку во время `processing` так же, как и другие источники входа.

Результат: снимок с камеры становится третьим источником входа в уже работающий пайплайн.

### Шаг 3. UX-детали и отказоустойчивость
Файлы: `lib/main.dart`, файл экрана камеры

1. Добавить понятные тексты ошибок:
   - разрешение на камеру отклонено;
   - камера недоступна на устройстве;
   - не удалось сделать снимок.
2. Убедиться, что после закрытия камеры пользователь всегда возвращается в предсказуемый state (`idle` или `processing`).
3. Для больших кадров при необходимости предусмотреть downscale/компрессию на этапе съемки (если будут OOM/долгая обработка на слабых устройствах).

Результат: стабильный пользовательский поток без зависаний и неясных состояний.

### Шаг 4. Тестирование и регрессия
Файлы: `test/widget_test.dart`, `integration_test/analyze_screen_test.dart` (+ ручные проверки)

1. Обновить UI-тесты на наличие новой кнопки `Сделать снимок`.
2. Сохранить существующие проверки локализованных ошибок и заголовка.
3. Добавить ручной чеклист Android-устройства/эмулятора:
   - открыть камеру -> изменить зум -> переключить вспышку -> сделать снимок;
   - убедиться, что экран камеры закрывается и стартует анализ;
   - проверить сценарий отказа в разрешении;
   - убедиться, что галерея/файлы продолжают работать.
4. Прогон: `flutter test` и `flutter analyze`.

Результат: новая функция проверена, критические регрессии исключены.

### Шаг 5. Обновление документации
Файл: `README.md`

1. Добавить новый пользовательский сценарий «Сделать снимок» в раздел flow.

Результат: актуальная документация по входным сценариям пайплайна.

## 5) Риски и меры снижения
- Риск: на части устройств/эмуляторов камера недоступна.
  - Мера: явный fallback-сценарий с понятной ошибкой и возвратом в `idle`.
- Риск: нестабильность flash/zoom в зависимости от аппаратуры.
  - Мера: читать capability контроллера (`min/max zoom`, поддерживаемые режимы flash) и скрывать/дизейблить недоступные контролы.
- Риск: большие фото замедляют анализ.
  - Мера: контролировать `ResolutionPreset`, при необходимости добавить downscale перед запуском пайплайна.
