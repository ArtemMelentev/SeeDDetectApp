# План: ограничение debug-изображений в релизе

## Контекст (текущее поведение)
1. Python пайплайн `segment_seeds_scan.py:analyze_image(...)` всегда пишет 4 артефакта в `output_dir`:
   - `<stem>_all_mask.png`
   - `<stem>_all_overlay.jpg`
   - `<stem>_black_mask.png`
   - `<stem>_black_overlay.jpg`
   И всегда возвращает их в `payload["artifacts"]`.
2. Flutter UI (`lib/main.dart`) отображает артефакты, читая ключи `all_mask/all_overlay/black_mask/black_overlay` и фильтруя пустые значения.
3. Android bridge (`android_bridge.py` + `MainActivity.kt`) просто прокидывает `input_path` и `runDir` в Python без понятия «релиз/дебаг».

## Требование
Только для релиз-сборки:
1. Не сохранять и не показывать никакие debug-изображения, кроме изображения с отмеченными черными семечками.
2. На изображении с черными семечками убрать пиксельную визуализацию маски и контуры.
3. Вместо этого отрисовывать по одному еле заметному красному кресту (или эквивалентному маркеру) на каждую детектированную черную семечку.
4. Если `black_seed_count == 0`, все равно сохранять один `black_overlay` (без крестиков), остальные изображения не создавать.
5. В debug/dev-сборках поведение не менять (текущие 4 артефакта и текущая визуализация должны сохраниться, чтобы не ломать тесты и отладку).

## Решение (архитектура переключения)
Сделать переключение режима на границе Android -> Python, используя флаг релизности сборки:
1. Kotlin (`MainActivity.kt`) определяет `isRelease = !BuildConfig.DEBUG`.
2. Kotlin передает этот флаг третьим аргументом в Python `android_bridge.run_analysis_json(...)`.
3. Python `android_bridge.py` принимает `release_mode` (опционально, с дефолтом), и прокидывает в `analyze_image(..., release_mode=...)`.
4. `segment_seeds_scan.py:analyze_image` добавляет опциональный параметр `release_mode: bool = False` (дефолт `False` сохраняет контракт тестов).

## Изменения по файлам

### 1) `android/app/src/main/kotlin/com/seeddetect/app/MainActivity.kt`
1. В `runAnalysis(...)` добавить вычисление `val isRelease = !BuildConfig.DEBUG`.
2. В `bridge.callAttr("run_analysis_json", ...)` передать третий параметр `isRelease`.

Ожидаемый эффект:
- В debug-сборке (`BuildConfig.DEBUG == true`) Python будет работать как сейчас.
- В release-сборке (`BuildConfig.DEBUG == false`) включится «санитайзинг» артефактов.

### 2) `android/app/src/main/python/android_bridge.py`
1. Изменить сигнатуру:
   - было: `def run_analysis_json(input_path, output_dir):`
   - станет: `def run_analysis_json(input_path, output_dir, release_mode=False):`
2. Вызов:
   - `result = analyze_image(input_path, output_dir, release_mode=bool(release_mode))`

### 3) `segment_seeds_scan.py`

#### 3.1. Параметр режима
1. Добавить параметр `release_mode: bool = False` в `analyze_image(...)`.
2. В debug-ветке оставить текущую логику записи 4 файлов и заполнения `payload["artifacts"]`.

#### 3.2. Релиз-политика артефактов
1. В релиз-ветке НЕ писать:
   - `*_all_mask.png`
   - `*_all_overlay.jpg`
   - `*_black_mask.png`
2. В релиз-ветке писать ТОЛЬКО:
   - `*_black_overlay.jpg`
     - всегда (даже если `black_seed_count == 0`)
     - содержимое: базовое изображение (как сейчас используется `normalized`) + крестики по центроидам контуров черных семечек.
3. В релиз-ветке возвращать `artifacts` только с одним ключом:
   - `{ "black_overlay": <path> }`
   Остальные ключи не включать (UI сам их отфильтрует как пустые).

#### 3.3. Новая визуализация для релиза
1. Добавить отдельный helper, например `_visualize_black_release(img_bgr: np.ndarray, black_contours: list) -> np.ndarray`.
2. Алгоритм:
   - `vis = img_bgr.copy()`
   - для каждого `cnt`:
     - вычислить центр через `cv2.moments(cnt)`; если `m00 == 0`, использовать центр `cv2.boundingRect(cnt)`
     - отрисовать маленький маркер фиксированного размера, НЕ зависящего от площади/маски:
       - предпочтительно `cv2.drawMarker(..., markerType=cv2.MARKER_TILTED_CROSS или cv2.MARKER_CROSS, markerSize=...)`
       - если `drawMarker` недоступен, fallback: две короткие диагональные линии (`cv2.line`) одинаковой длины.
   - не рисовать заливку маской, контуры, номера, текстовые блоки.
3. Размер/толщина маркера:
   - «еле заметный»: `thickness=1`, цвет `(0,0,255)`.
   - `markerSize` можно выбрать слабозависимым от размера изображения, но НЕ от размера семечки, например:
     - `markerSize = max(8, int(round(min(h, w) * 0.008)))` с верхним ограничением (например `<= 18`).

## Совместимость и риски
1. Python unit-тесты (`tests/test_segment_contract.py`) ожидают 4 артефакта. Поэтому дефолтный режим `release_mode=False` должен сохранить текущее поведение.
2. Изменение сигнатуры `run_analysis_json` безопасно, так как Kotlin начнет передавать 3-й аргумент, а другие вызовы отсутствуют.
3. Flutter UI не нужно менять: он уже отфильтровывает пустые/отсутствующие артефакты и покажет только `black_overlay`.

## Проверка (что сделать после внедрения)
1. Локально (Python): запустить `python -m unittest -q` из корня репозитория, убедиться что тесты проходят.
2. Android debug build:
   - выполнить анализ изображения и убедиться, что создаются 4 файла и визуализация прежняя.
3. Android release build:
   - выполнить анализ и убедиться:
     - создается только `*_black_overlay.jpg`
     - на нем только маленькие крестики (или он пустой, если `black_seed_count == 0`)
     - отсутствуют `*_all_*` и `*_black_mask.png`

## Definition of Done
1. В release-сборке пайплайн сохраняет только один артефакт `black_overlay`.
2. `black_overlay` не содержит пиксельной заливки/контуров/текста; только маленькие крестики по одному на каждую черную семечку.
3. В debug/dev режимах поведение и тесты не регрессируют.
