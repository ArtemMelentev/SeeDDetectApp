# План: Android-приложение с локальным запуском Python и заделом под iOS

## 1) Зафиксированные требования
- Платформа v1: `Android`.
- Минимальная версия Android: `API 29 (Android 10+)`.
- Обработка: `локально на устройстве`, без интернета.
- Python-скрипт: `segment_seeds_scan.py`.
- UI: минимальный, с кнопкой выбора файла.
- Источники файлов: галерея и файловая система.
- Режим обработки v1: `один файл за запуск` (без batch).
- Вывод пользователю: метрики + оверлеи/маски.
- Хранение на сервере: не требуется.
- Ограничение на размер файла: жесткого лимита нет (нужна защита от OOM через downscale/потоковую обработку).

## 2) Рекомендуемый стек под текущие ограничения

### Мобильный клиент
- `Flutter` для UI и навигации (сразу готовая база под будущий iOS-UI).

### Исполнение Python на Android
- Нативный Android-модуль в проекте Flutter (`Kotlin`) + `Chaquopy` для запуска Python.
- Передача данных из Flutter в нативный слой через `Platform Channel`.

### Почему именно так
- Выполняется требование полного офлайна на Android.
- Быстрое внедрение существующего Python-кода без переписывания алгоритма.
- Flutter-часть (UI/логика сценария) останется общей при переходе на iOS.

## 3) Ключевое ограничение по iOS
- Python + Chaquopy решает задачу только для Android.
- Для iOS потребуется отдельная реализация вычислительного ядра (наиболее практично: перенос алгоритма в `C++/OpenCV` и вызов из Android/iOS, либо iOS-специфичный модуль).
- Поэтому план строится как двухэтапный: сначала рабочий Android офлайн, затем перенос ядра для iOS.

## 4) План реализации Android v1 (офлайн)

### Этап A. Подготовка Python-скрипта как модуля
1. Рефактор `segment_seeds_scan.py` в библиотечный режим:
   - убрать жестко заданные пути из `main()`;
   - оставить чистую точку входа: `analyze_image(input_path, output_dir) -> dict`;
   - возвращать структурированный JSON + пути к созданным артефактам.
2. Унифицировать ответ:
   - метрики: `seed_count`, `all_seed_area_px`, `black_seed_count`, `black_seed_area_px`;
   - служебные поля: `image_w`, `image_h`, `processing_ms`;
   - артефакты: пути к `*_all_mask.png`, `*_all_overlay.jpg`, `*_black_mask.png`, `*_black_overlay.jpg`.
3. Добавить обработку ошибок:
   - неподдерживаемый формат/битый файл;
   - исключения OpenCV;
   - корректный JSON ошибки для UI.

### Этап B. Интеграция Python в Android слой
1. Подключить `Chaquopy` в Android-модуль Flutter-проекта.
2. Добавить Python-зависимости (`opencv-python`/совместимый пакет, `numpy`) с учетом ограничений мобильной сборки.
3. Создать Kotlin-сервис-обертку:
   - принимает URI выбранного файла;
   - копирует файл во внутреннее хранилище приложения;
   - вызывает Python-функцию;
   - получает JSON и пути результатов;
   - отдает данные в Flutter.

### Этап C. Flutter UI
1. Экран `AnalyzeScreen`:
   - кнопка выбора файла;
   - индикатор выполнения;
   - вывод численных метрик;
   - отображение 2-4 изображений результатов.
2. Выбор файла:
   - поддержка галереи и системного picker (`image_picker` + `file_picker` или единый picker).
3. Состояния:
   - `idle` -> `picking` -> `processing` -> `done/error`.
4. UX ошибок:
   - понятные сообщения при плохом файле, слишком большом размере, сбое обработки.

### Этап D. Android разрешения и хранение
1. Настроить разрешения и scoped storage под актуальный Android API.
2. Все временные результаты хранить во внутренней директории приложения.
3. Добавить очистку временных файлов после показа результата (или по TTL).

### Этап E. Тестирование и приемка
1. Unit/integration-тесты для Python-обертки на 2-3 эталонных изображениях.
2. Инструментальные тесты Android/Flutter на сценарий выбора и обработки.
3. Критерии готовности:
   - файл выбирается из галереи и из файлов;
   - скрипт стабильно запускается локально;
   - пользователь видит метрики и оверлеи;
   - офлайн-режим полностью рабочий.

## 5) План миграции к iOS (этап v2)
1. Заморозить контракт ядра обработки (вход/выход JSON и набор артефактов).
2. Перенести алгоритм из Python в кроссплатформенное ядро (`C++/OpenCV`) с одинаковым API.
3. Подключить это ядро через платформенные мосты Android+iOS, Flutter UI оставить тем же.
4. Провести сверку точности: сравнение результатов C++-версии с текущим Python на референсных изображениях.

## 6) Риски и меры
- Риск: мобильная совместимость Python-зависимостей (особенно OpenCV).
  - Мера: ранний PoC только запуска скрипта на реальном Android-устройстве.
- Риск: медленная обработка больших изображений.
  - Мера: предварительный resize + профилирование + индикатор прогресса.
- Риск: iOS офлайн не покрывается текущим Python-рантаймом.
  - Мера: плановый перенос ядра в C++ на этапе v2.

## 7) Что получим после реализации этого плана
- Рабочее Android-приложение офлайн, запускающее текущий Python-скрипт и показывающее метрики + оверлеи.
- Архитектурный задел для iOS на уровне Flutter UI и стабильного контракта обработки.

## 9) Принятые допущения для старта реализации
- Реализацию начинаем с Android-only офлайн архитектуры (`Flutter + Kotlin + Chaquopy`).
- Для больших изображений применяем безопасную предобработку (resize до рабочего разрешения) перед запуском алгоритма, чтобы избежать OOM.
- Контракт входа/выхода ядра фиксируем с первого этапа, чтобы упростить перенос вычислительной части на iOS в v2.
