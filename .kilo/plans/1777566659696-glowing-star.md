# План: починить VS Code tasks для Android (эмулятор + APK)

## Цель
- Восстановить работоспособность двух задач:
  - `Start SeedDetect emulator`
  - `Build SeedDetect release APK (phone)`
- Убрать зависимость от путей конкретного пользователя/ПК.

## Что уже диагностировано
- В `.vscode/tasks.json` захардкожены пути, которые сейчас не существуют:
  - `C:\Android\Sdk\...`
  - `C:\Users\Artem Melentyev\source\flutter\bin\flutter.bat`
- В `.vscode/launch.json` и `.vscode/settings.json` также используются эти абсолютные пути.
- `android/local.properties` в репозитории отсутствует (типично, т.к. часто в `.gitignore`), поэтому резолв Flutter/SDK должен иметь fallback-логику.
- Текущая release-сборка уже требует signing (`android/app/build.gradle`), значит задача должна давать понятную ошибку, если `android/key.properties` не настроен.

## План изменений

### 1) Переписать `Start SeedDetect emulator` на переносимую логику
Файл: `.vscode/tasks.json`
- Зафиксировать shell задач как Windows PowerShell (`powershell.exe`) для предсказуемого исполнения.
- Внутри task:
  - определить Android SDK путь приоритетно из `ANDROID_SDK_ROOT`/`ANDROID_HOME`;
  - fallback: попытка найти `adb`/`emulator` через `Get-Command`;
  - если не найдено — завершать с понятной диагностикой.
- Проверять доступность AVD `SeedDetect_API34`; если нет — fallback на первый доступный AVD с явным логом выбора.
- Корректно обрабатывать кейс уже запущенного эмулятора (не стартовать второй экземпляр).
- Оставить ожидание загрузки (`sys.boot_completed`) и разблокировку экрана.

### 2) Починить `Run SeedDetect debug (emulator)`
Файл: `.vscode/tasks.json`
- Сохранить `dependsOn: Start SeedDetect emulator`.
- Резолвить `flutter` так:
  1. из `android/local.properties` (`flutter.sdk`) если файл есть,
  2. иначе из `Get-Command flutter`/`flutter.bat`.
- Перед запуском выставлять `ANDROID_HOME`/`ANDROID_SDK_ROOT` только если путь реально найден.
- Запускать `flutter run -d emulator-5554 --device-timeout 120` через найденный бинарник.

### 3) Починить `Build SeedDetect release APK (phone)`
Файл: `.vscode/tasks.json`
- Использовать ту же переносимую логику резолва `flutter` и Android SDK.
- Убрать безусловный `flutter clean` из основной release-задачи (ускорение и меньше побочных эффектов).
- Оставить `flutter pub get` -> `flutter build apk --release --target-platform android-arm64`.
- Добавить pre-check для `android/key.properties` с понятным текстом ошибки (чтобы не падать неявно глубоко в Gradle).

### 4) Синхронизировать debug-конфиги VS Code
Файл: `.vscode/launch.json`
- Убрать захардкоженный `flutter.bat` из `SeedDetect (One Click)`.
- Перевести `SeedDetect (One Click)` на запуск через починенную задачу `Run SeedDetect debug (emulator)`.
- Оставить `preLaunchTask` на старт эмулятора там, где это нужно.

### 5) Снизить хрупкость workspace-настроек
Файл: `.vscode/settings.json`
- Убрать/минимизировать абсолютный `dart.flutterSdkPath`, если он мешает переносимости.
- Сохранить только те env-переменные, которые безопасно оставлять как дефолт, без привязки к конкретному пользователю.

## Проверка после внедрения
- Task `Start SeedDetect emulator`:
  - эмулятор стартует или переиспользуется,
  - нет ошибки boot timeout,
  - нет зависимости от старого пути `C:\Android\Sdk`.
- Task `Run SeedDetect debug (emulator)`:
  - приложение запускается на `emulator-5554`.
- Task `Build SeedDetect release APK (phone)`:
  - при валидном signing появляется `build/app/outputs/flutter-apk/app-release.apk`;
  - при отсутствии signing выводится понятная инструкция про `android/key.properties`.

## Риски и меры
- Несколько эмуляторов/устройств в `adb`: фиксируем целевой serial и явный выбор AVD.
- Отсутствие `local.properties` на чистой машине: делаем fallback на `Get-Command flutter`.
- Различный shell в VS Code: принудительно задаем PowerShell в задачах.
