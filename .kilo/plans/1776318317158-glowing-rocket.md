# План: релиз SeedDetect для установки на Android-телефон

## Цель
Собрать и отдать установочный Android-релиз (APK) для физического телефона, чтобы приложение запускалось вне эмулятора и выполняло локальную обработку изображений.

## Текущее состояние (по коду репозитория)
- Проект уже Android-first: Flutter + Kotlin bridge + Chaquopy + Python/OpenCV (`README.md`, `android/app/build.gradle`).
- В `android/app/build.gradle` сейчас жестко задано `abiFilters "x86_64"` — это блокирует запуск на большинстве телефонов (обычно `arm64-v8a`).
- Для `release` сейчас используется `signingConfigs.debug`, то есть нет нормальной release-подписи.
- Сетевой риск Chaquopy уже фиксировался ранее: таймауты при скачивании wheel из `chaquo.com`.

## Принятая стратегия релиза
- Формат поставки: **signed release APK для sideload на телефон** (рекомендованный и самый быстрый путь под текущую задачу).
- Целевая архитектура релиза: **arm64-v8a**.
- Обязательное ограничение: **эмуляторный debug-режим (x86_64) должен остаться рабочим после изменений**.
- Итоговый артефакт: `build/app/outputs/flutter-apk/app-arm64-v8a-release.apk`.

## План реализации

### Этап 1. Подготовить release-подпись (без коммита секретов)
1. Сгенерировать keystore локально (например, `android/app/upload-keystore.jks`).
2. Создать `android/key.properties` (storeFile/storePassword/keyAlias/keyPassword).
3. Проверить, что секреты не попадут в git (в `.gitignore` уже есть `**/android/key.properties` и `*.jks`).

Ожидаемый результат: локально есть рабочая release-подпись, но секреты не хранятся в репозитории.

### Этап 2. Исправить Android-конфигурацию под телефонный релиз
Изменения в `android/app/build.gradle`:
1. Подключить чтение `key.properties` и объявить `signingConfigs.release`.
2. Перевести `buildTypes.release` на `signingConfigs.release`.
3. Перенастроить ABI так, чтобы одновременно соблюдались два сценария:
   - debug/эмулятор продолжает работать на `x86_64`;
   - release для телефона собирается под `arm64-v8a`.
4. Техническая реализация ABI выбрать без деградации текущего run-пути (`flutter run` на эмуляторе):
   - рекомендованный безопасный вариант: `abiFilters "x86_64", "arm64-v8a"` + сборка release с `--split-per-abi` и выдача только arm64 APK пользователю;
   - альтернативный вариант: развести ABI по variant/flavor (если потребуется минимизация размера/загрузок).
5. Сохранить совместимость Chaquopy-конфига (`Python 3.10`, `numpy`, `opencv-python-headless`) и проверить, что разрешение ABI согласовано с pip-артефактами.

Ожидаемый результат: release-вариант собирается под arm64 и подписывается release-ключом.

### Этап 3. Собрать релизный APK
1. Выполнить сборку:
   - `flutter clean`
   - `flutter pub get`
   - `flutter build apk --release --split-per-abi`
2. Убедиться, что появился arm64-артефакт:
   - `build/app/outputs/flutter-apk/app-arm64-v8a-release.apk`

Ожидаемый результат: готовый installable APK под телефон.

### Этап 4. Проверить установку и запуск на физическом устройстве
1. Установить APK через ADB (`adb install -r ...`) или вручную.
2. На телефоне открыть приложение и пройти smoke-тест:
   - выбрать изображение из галереи/файлов;
   - дождаться обработки;
   - убедиться, что отображаются метрики и артефакты.
3. Если раньше на телефоне стояла debug-сборка с тем же `applicationId`, удалить ее перед установкой release (из-за несовпадения подписи).

### Этап 4.1. Регрессия эмулятора (обязательная)
1. После изменений выполнить запуск debug на эмуляторе (`flutter run`).
2. Пройти минимальный smoke на эмуляторе: выбор файла -> обработка -> отображение метрик/артефактов.
3. Зафиксировать, что переход к телефонному релизу не сломал текущий x86_64 цикл разработки.

Ожидаемый результат: приложение стабильно запускается на телефоне и в эмуляторе, выполняя основной сценарий в обоих режимах.

### Этап 5. Зафиксировать инструкцию релиза
1. Обновить `README.md` кратким runbook:
   - как собирать arm64 release APK;
   - где лежит артефакт;
   - как установить на телефон.
2. Добавить секцию troubleshooting (минимум 2 пункта):
   - Chaquopy timeout;
   - отсутствие wheel для выбранной комбинации ABI/Python/package version.

Ожидаемый результат: сборка и установка воспроизводимы без ручного расследования.

## Риски и fallback
- **Риск 1: Chaquopy timeout до `chaquo.com`.**
  - Fallback: повтор с устойчивой сетью/VPN, затем настройка локального wheel-кэша/`--find-links`.
- **Риск 2: недоступен arm64 wheel для pinned версии `opencv-python-headless`.**
  - Fallback: подобрать совместимую версию из списка доступных для текущих `chaquopy + python + abi`.
- **Риск 3: конфликт подписи при переустановке поверх debug.**
  - Fallback: удалить установленный пакет и поставить release заново.

## Definition of Done
- В `android/app/build.gradle` release-подпись настроена корректно и не использует debug key.
- Эмуляторный debug-сценарий (`x86_64`) по-прежнему рабочий после всех изменений.
- Сборка `flutter build apk --release --split-per-abi` завершается успешно.
- Получен артефакт `app-arm64-v8a-release.apk`.
- APK устанавливается на физический Android-телефон и проходит smoke-сценарий обработки изображения.
- README содержит короткую рабочую инструкцию по повторной сборке и установке.
