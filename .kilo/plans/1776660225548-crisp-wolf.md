# План внедрения: Google Drive «Отправить копию» (PDF/JPEG)

## 1) Цель и критерий готовности
- Цель: приложение появляется в Google Drive в сценарии **«Отправить копию»** и корректно принимает `application/pdf` и `image/*`.
- Критерий готовности (Definition of Done):
  - приложение видно в Android Sharesheet для PDF и JPEG из Google Drive;
  - файл открывается внутри приложения после выбора приложения в шаринге;
  - поддержаны `ACTION_SEND` (обязательно) и `ACTION_SEND_MULTIPLE` (желательно, но включаем сразу);
  - обработаны ошибки чтения `content://` URI;
  - пройдены обязательные ручные тесты.

## 2) Границы внедрения
- Включаем: манифест, прием интентов, чтение URI через `ContentResolver`, импорт/копирование во внутреннее хранилище, базовый UX, логирование, тесты.
- Не включаем (после MVP): расширенная история файлов, сложный импорт-центр, дополнительные форматы кроме PDF/JPEG.

## 3) Пошаговый план реализации

### Шаг 0. Предпроектная проверка текущей реализации
1. Найти текущую точку входа для открытия документов (существующая `Activity`/`DeepLink`/`Viewer`).
2. Проверить `AndroidManifest.xml`: существующие `intent-filter`, `action`, `category`, MIME-типы, `android:exported`.
3. Зафиксировать, можно ли переиспользовать существующую `Activity` или нужна отдельная `ShareReceiverActivity`.

Результат шага: подтверждено текущее состояние и выбран целевой обработчик шаринга.

### Шаг 1. Обновление AndroidManifest под Google Drive Share
1. Добавить (или обновить) `Activity`-приемник с `android:exported="true"`.
2. Добавить отдельные `intent-filter` для:
   - `android.intent.action.SEND` + `application/pdf`;
   - `android.intent.action.SEND` + `image/*`;
   - `android.intent.action.SEND_MULTIPLE` + `application/pdf`;
   - `android.intent.action.SEND_MULTIPLE` + `image/*`.
3. В каждый фильтр добавить `android.intent.category.DEFAULT`.
4. Проверить отсутствие конфликтующих фильтров, которые перехватывают не те MIME-типы.

Результат шага: приложение матчится Android Sharesheet для нужных MIME-сценариев.

### Шаг 2. Реализация приема интента в ShareReceiverActivity
1. В `onCreate`/`onNewIntent` разобрать `intent.action`.
2. Для `ACTION_SEND`:
   - получить `Uri` из `Intent.EXTRA_STREAM`;
   - получить `intent.type`;
   - передать в единый обработчик импорта.
3. Для `ACTION_SEND_MULTIPLE`:
   - получить список `Uri` из `Intent.EXTRA_STREAM`;
   - валидировать непустой список;
   - обработать каждый URI через тот же обработчик.
4. Для неподдерживаемого action — безопасное завершение с понятной ошибкой.

Результат шага: интенты шаринга стабильно попадают в единый pipeline импорта.

### Шаг 3. Единый модуль импорта из content:// URI
1. Создать/доработать сервис импорта (например, `SharedContentImporter`).
2. Реализовать чтение только через `ContentResolver.openInputStream(uri)`.
3. Собирать метаданные:
   - MIME (`ContentResolver.getType` + fallback из `intent.type`);
   - отображаемое имя (`OpenableColumns.DISPLAY_NAME`);
   - размер (`OpenableColumns.SIZE`, если есть).
4. Копировать поток во внутреннее хранилище (`cacheDir`/`filesDir`) с безопасным именованием.
5. Валидировать поддерживаемые типы: `application/pdf` и `image/*`.
6. Возвращать структурированный результат: success/error + причина ошибки.

Результат шага: импорт не зависит от «файлового пути», работает с `content://` из Drive.

### Шаг 4. Интеграция с UX после импорта
1. MVP-сценарий:
   - импорт успешен -> открыть экран просмотра документа/изображения;
   - импорт неуспешен -> показать понятное сообщение и завершить экран приемника.
2. Если файлов несколько (`SEND_MULTIPLE`):
   - для MVP открыть первый поддерживаемый файл;
   - остальные сохранить и отразить количеством/статусом (лог + уведомление в UI).
3. Убедиться, что `ShareReceiverActivity` не остается лишним в back stack (корректный `finish`).

Результат шага: предсказуемый пользовательский поток после выбора приложения в Drive.

### Шаг 5. Логирование и диагностика
1. Добавить структурированные логи:
   - входящий `action`;
   - MIME (`intent.type` и итоговый MIME);
   - URI (без утечки чувствительных данных);
   - факт открытия `InputStream`;
   - результат копирования/ошибки.
2. Для ошибок — лог с причиной (`SecurityException`, `FileNotFoundException`, `IOException`, unsupported MIME).
3. Добавить защиту от падений: локальные `try/catch` вокруг I/O с контролируемым fallback.

Результат шага: проблемы воспроизводимы и быстро диагностируются.

### Шаг 6. Тестирование (ручное + при наличии авто)
1. Обязательные ручные сценарии:
   - Google Drive -> «Отправить копию» -> PDF -> приложение видно -> файл открыт.
   - Google Drive -> «Отправить копию» -> JPEG -> приложение видно -> файл открыт.
2. Дополнительные:
   - `SEND_MULTIPLE` для нескольких изображений;
   - большой файл;
   - файл без расширения, но с корректным MIME;
   - отправка из Files/Gmail/Telegram/файлового менеджера.
3. Негативные:
   - неподдерживаемый MIME;
   - поврежденный/недоступный URI;
   - ошибка чтения потока.
4. Проверка регрессий:
   - существующее открытие документов внутри приложения не сломано;
   - переустановка приложения после изменения манифеста.

Результат шага: подтверждена работоспособность и отсутствие критичных регрессий.

### Шаг 7. Приемка и выпуск
1. Сверить DoD и закрыть checklist P0/P1.
2. Подготовить краткий отчет внедрения:
   - измененные файлы;
   - принятые UX-решения;
   - результаты тестов (pass/fail по сценариям);
   - известные ограничения MVP.
3. Передать в релизный контур (внутренний QA/стейджинг/прод в рамках текущего процесса команды).

Результат шага: фича готова к использованию и сопровождаемости.

## 4) Приоритеты выполнения
- P0 (обязательно): манифест (`SEND` + MIME + `exported`), импорт через `ContentResolver`, открытие файла в приложении.
- P1 (в этой же итерации): `SEND_MULTIPLE`, устойчивые ошибки, структурированное логирование.
- P2 (после MVP): расширенный импорт-экран, история, дополнительные форматы.

## 5) Риски и меры снижения
- Риск: приложение не появляется в шаринге -> проверить точные MIME, `CATEGORY_DEFAULT`, `exported`, переустановку после правок манифеста.
- Риск: URI не читается -> fallback/ошибки на уровне импортера, явное сообщение пользователю.
- Риск: расхождение MIME (`image/jpeg` vs `image/*`) -> использовать `image/*` в фильтрах и валидации.
- Риск: конфликт с существующими интентами -> изолировать обработку в отдельной `ShareReceiverActivity`.

## 6) Артефакты по завершении
- Обновленный `AndroidManifest.xml`.
- Код `ShareReceiverActivity` (или эквивалентного обработчика).
- Код импортера `content://` URI в локальное хранилище.
- Описание конечного UX после получения файла.
- Отчет по тест-сценариям Google Drive share.
