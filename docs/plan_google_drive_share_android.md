# План для агента: добавить открытие PDF/JPEG из Google Drive через «Отправить копию»

## Цель
Сделать так, чтобы Android-приложение появлялось в Google Drive в меню **«Отправить копию»** и могло принимать **PDF** и **JPEG** документы для дальнейшего открытия внутри приложения.

---

## Что должно получиться в итоге
Пользователь в Google Drive:
1. открывает файл PDF или JPEG;
2. нажимает **три точки**;
3. выбирает **«Отправить копию»**;
4. видит среди доступных приложений **наше приложение**;
5. выбирает его;
6. файл открывается или импортируется внутри приложения.

---

## Гипотезы в порядке возможной полезности

### Гипотеза 1. Основная и наиболее вероятная причина
Приложение не появляется в списке, потому что в нем **не объявлен корректный `intent-filter`** для приема файлов через системный механизм шаринга Android.

**Что проверить:**
- есть ли `Activity`, принимающая `android.intent.action.SEND`;
- есть ли `CATEGORY_DEFAULT`;
- указаны ли нужные MIME-типы:
  - `application/pdf`
  - `image/*` или `image/jpeg`;
- выставлен ли `android:exported="true"` у `Activity` с `intent-filter`.

**Ожидаемый эффект:**  
После добавления корректных `intent-filter` приложение начнет отображаться в меню **«Отправить копию»** в Google Drive.

---

### Гипотеза 2. Приложение принимает не тот тип интента
Даже если приложение умеет открывать документы, оно может быть настроено только на `ACTION_VIEW`, а Google Drive в сценарии **«Отправить копию»** использует именно **share-механику**, то есть `ACTION_SEND` и иногда `ACTION_SEND_MULTIPLE`.

**Что сделать:**
- добавить поддержку `ACTION_SEND`;
- при необходимости добавить `ACTION_SEND_MULTIPLE`.

**Ожидаемый эффект:**  
Приложение станет совместимым именно с тем сценарием, который использует Google Drive.

---

### Гипотеза 3. MIME-типы описаны слишком узко или неверно
Если приложение ожидает только `image/jpeg`, а Google Drive отправляет `image/*`, либо наоборот, Android не покажет приложение в списке.

**Что сделать:**
- для изображений использовать более широкий тип `image/*`;
- для PDF оставить точный тип `application/pdf`;
- при необходимости сделать отдельные `intent-filter` под разные типы.

**Ожидаемый эффект:**  
Повысится вероятность корректного матчинга интента и появления приложения в списке.

---

### Гипотеза 4. Файл приходит как `content://Uri`, а приложение не умеет его читать
Даже если приложение появится в списке, оно может не открыть файл, если код ожидает путь к файлу, а не `content://` URI.

**Что сделать:**
- принимать URI из `Intent.EXTRA_STREAM`;
- читать файл через `ContentResolver.openInputStream(uri)`;
- при необходимости копировать файл во внутреннее хранилище приложения.

**Ожидаемый эффект:**  
Файл из Google Drive будет стабильно открываться внутри приложения.

---

### Гипотеза 5. Проблема в сборке или установке, а не в логике
Иногда `intent-filter` уже добавлен, но приложение все равно не показывается, потому что:
- установлен старый билд;
- приложение не было переустановлено после изменения манифеста;
- тестируют не ту `Activity`;
- есть конфликт фильтров.

**Что сделать:**
- удалить приложение с устройства;
- поставить свежую сборку;
- проверить итоговый `AndroidManifest`;
- протестировать на реальном устройстве.

**Ожидаемый эффект:**  
Исключение ложных причин, связанных с окружением.

---

## План реализации для агента

### Этап 1. Проверка текущей конфигурации приложения
Нужно:
1. Найти `Activity`, которая должна принимать документ из внешних приложений.
2. Проверить текущий `AndroidManifest.xml`.
3. Зафиксировать:
   - есть ли `intent-filter`;
   - какие `action` уже поддерживаются;
   - какие MIME-типы указаны;
   - выставлен ли `android:exported`.

**Результат этапа:**  
Понимание, почему приложение сейчас не попадает в Android Sharesheet.

---

### Этап 2. Добавить поддержку получения файлов из Google Drive
Нужно создать или настроить отдельную `Activity`, например `ShareReceiverActivity`, которая будет принимать входящие файлы.

Пример конфигурации в `AndroidManifest.xml`:

```xml
<activity
    android:name=".ShareReceiverActivity"
    android:exported="true">

    <intent-filter>
        <action android:name="android.intent.action.SEND" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:mimeType="application/pdf" />
    </intent-filter>

    <intent-filter>
        <action android:name="android.intent.action.SEND" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:mimeType="image/*" />
    </intent-filter>

    <intent-filter>
        <action android:name="android.intent.action.SEND_MULTIPLE" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:mimeType="application/pdf" />
    </intent-filter>

    <intent-filter>
        <action android:name="android.intent.action.SEND_MULTIPLE" />
        <category android:name="android.intent.category.DEFAULT" />
        <data android:mimeType="image/*" />
    </intent-filter>
</activity>
```

**Важно:**
- MIME-типы писать в нижнем регистре;
- `android:exported="true"` обязателен;
- лучше делать отдельные `intent-filter`, а не один общий со всем сразу.

**Результат этапа:**  
Приложение должно начать отображаться в списке приложений для «Отправить копию».

---

### Этап 3. Реализовать прием входящего файла
В `ShareReceiverActivity` нужно обработать `Intent`.

Логика:
1. Проверить `intent.action`.
2. Если это `ACTION_SEND`:
   - взять `Uri` из `Intent.EXTRA_STREAM`;
   - определить MIME-тип;
   - передать файл в слой открытия/импорта.
3. Если это `ACTION_SEND_MULTIPLE`:
   - получить список `Uri`;
   - обработать каждый.

Пример на Kotlin:

```kotlin
class ShareReceiverActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        when (intent?.action) {
            Intent.ACTION_SEND -> {
                val uri = intent.getParcelableExtra<Uri>(Intent.EXTRA_STREAM)
                val type = intent.type
                if (uri != null) {
                    handleSharedUri(uri, type)
                } else {
                    finishWithError("URI отсутствует")
                }
            }

            Intent.ACTION_SEND_MULTIPLE -> {
                val uris = intent.getParcelableArrayListExtra<Uri>(Intent.EXTRA_STREAM)
                val type = intent.type
                if (!uris.isNullOrEmpty()) {
                    uris.forEach { handleSharedUri(it, type) }
                } else {
                    finishWithError("Список URI пуст")
                }
            }

            else -> {
                finishWithError("Неподдерживаемый action")
            }
        }
    }

    private fun handleSharedUri(uri: Uri, mimeType: String?) {
        contentResolver.openInputStream(uri)?.use { input ->
            // Скопировать файл во внутреннее хранилище
            // либо сразу передать на экран просмотра / импорта
        }
    }

    private fun finishWithError(message: String) {
        // Логирование + показ ошибки пользователю при необходимости
        finish()
    }
}
```

**Результат этапа:**  
Приложение не просто появится в списке, но и сможет реально принять файл.

---

### Этап 4. Нормализовать работу с файлами
Так как из Google Drive обычно приходит `content://Uri`, нельзя полагаться на «реальный путь к файлу».

Нужно:
1. Сделать универсальный метод чтения через `ContentResolver`.
2. Получать:
   - имя файла;
   - MIME-тип;
   - размер, если доступен.
3. Копировать файл во внутренний кеш или постоянное хранилище приложения.
4. Уже оттуда открывать его в viewer или передавать дальше в бизнес-логику.

**Рекомендуемый подход:**
- не работать с абсолютным путем;
- работать только через URI и поток;
- хранить импортированную копию у себя.

**Результат этапа:**  
Повышение стабильности и совместимости с Google Drive и другими источниками файлов.

---

### Этап 5. Определить UX-сценарий после получения файла
Агенту нужно решить, что делать после получения файла:
- сразу открывать экран просмотра документа;
- показывать экран импорта;
- сохранять файл в локальную коллекцию приложения;
- предлагать выбрать действие.

**Минимально жизнеспособный сценарий:**
1. Получили файл;
2. скопировали локально;
3. открыли экран просмотра;
4. если формат не поддерживается — показать понятную ошибку.

---

### Этап 6. Добавить диагностику и логирование
Нужно заложить базовую диагностику:
- лог входящего `action`;
- лог `intent.type`;
- лог URI;
- лог успешного открытия потока;
- лог ошибок чтения файла.

Это сильно упростит отладку, если приложение появится в списке, но файл не будет открываться.

---

### Этап 7. Провести тестирование
Сценарии тестирования:

#### Обязательные
1. Отправка PDF из Google Drive.
2. Отправка JPEG из Google Drive.
3. Проверка появления приложения в меню **«Отправить копию»**.
4. Проверка открытия файла после выбора приложения.

#### Дополнительные
5. Отправка нескольких изображений.
6. Отправка файла большого размера.
7. Отправка файла без расширения, но с MIME.
8. Отправка из других приложений:
   - Files;
   - Gmail;
   - Telegram;
   - файловый менеджер.

#### Негативные

10. Приложение не может прочитать поток.
11. MIME-тип не поддерживается.

**Критерий приемки:**  
Пользователь может из Google Drive через **«Отправить копию»** передать PDF/JPEG в приложение, и файл корректно открывается внутри него.

## Приоритет реализации

### P0 — обязательно
- Добавить `intent-filter` для `ACTION_SEND`;
- добавить MIME-типы `application/pdf` и `image/*`;
- сделать `android:exported="true"`;
- реализовать чтение `Uri` через `ContentResolver`.

### P1 — очень желательно
- Поддержать `ACTION_SEND_MULTIPLE`;
- копировать файл во внутреннее хранилище;
- добавить логирование и обработку ошибок.

### P2 — можно после базовой версии
- Красивый экран импорта;
- история открытых файлов;
- поддержка дополнительных форматов;
- выбор сценария после импорта.

## Риски и типовые ошибки

1. Добавлен только `ACTION_VIEW`, но не `ACTION_SEND`.
2. Нет `CATEGORY_DEFAULT`.
3. Указан слишком узкий MIME-тип.
4. Нет `android:exported="true"`.
5. Код ждет файловый путь вместо `content://Uri`.
6. Приложение не переустановлено после изменения манифеста.
7. Нет обработки ошибок чтения потока.
