## План оставшихся работ

## Сессия запуска (VPN включен)

### Шаг 1: быстрый re-check окружения
- Проверить `flutter --version`, `flutter doctor -v`, `py -3.10 --version`.
- Проверить, что эмулятор виден в `flutter devices`.

Критерий успеха: toolchain green и есть `emulator-5554` (или аналогичный Android device).

### Шаг 2: повторная сборка с детальным логом
- Запустить `flutter run -v` (или `android\\gradlew.bat app:assembleDebug --stacktrace --info`).
- Если сборка пошла, дождаться полной установки приложения.
- Если снова timeout на Chaquopy, сохранить блок лога с URL, пакетом и типом сетевой ошибки.

Критерий успеха: debug APK/AAB собирается и ставится без сетевого timeout.

### Шаг 3: smoke-проверка функционала
- Открыть приложение на эмуляторе.
- Загрузить одно тестовое изображение через `Select from files`.
- Проверить отображение метрик и артефактов.

Критерий успеха: сценарий обработки завершается без ошибки, UI показывает результат.

### Шаг 4: фиксация результата
- Если запуск успешен: обновить `docs/setup-summary.md` фактическим статусом и командой запуска.
- Если запуск неуспешен: зафиксировать точный новый блокер (1 причина + 1 следующий шаг).

Критерий успеха: после сессии есть однозначный статус "работает" или "блокер X, делаем Y".

### Этап A: точная диагностика блокера Chaquopy
- Снять полный лог неуспешной сборки (`flutter run -v` или `android\\gradlew.bat app:assembleDebug --stacktrace --info`).
- Выделить из лога:
  - точный пакет/wheel, на котором падает загрузка;
  - тип ошибки (connect timeout, TLS/cert, proxy/firewall, DNS, 403/429);
  - повторяемость (разовая просадка сети или стабильный сбой).
- Отдельно проверить доступность `https://chaquo.com/pypi-13.1` из той же сети/сессии.

Критерий выхода этапа: есть зафиксированная первопричина, а не общий симптом timeout.

### Этап B: устранение сетевого блокера (основной путь + fallback)
- Основной путь (приоритет): обеспечить стабильный доступ к `chaquo.com` (прокси/VPN/исключение в firewall), затем повторить сборку.
- Fallback при нестабильной сети:
  - подготовить локальный wheelhouse/кэш требуемых пакетов Chaquopy;
  - подключить локальный источник в `chaquopy { pip { ... } }` через `--find-links`/локальный индекс;
  - при необходимости закрепить версии, точно доступные для `Python 3.10 + x86_64`.
- Проверка фикса: `:app:assembleDebug` завершается без timeout.

Критерий выхода этапа: сборка Android-пакета стабильно проходит минимум 2 раза подряд.

### Этап C: полный запуск и функциональная приемка
- Запустить `flutter run` на `SeedDetect_API34`.
- Проверить, что приложение стартует без падений.
- Прогнать пользовательский сценарий:
  - добавить тестовое изображение в эмулятор (drag-and-drop или `adb push`);
  - обработать через `Select from files`;
  - убедиться в наличии метрик и 2-4 артефактов (mask/overlay).
- Проверить, что артефакты реально создаются в cache (`seed_runs/run_*`).

Критерий выхода этапа: end-to-end сценарий успешно завершен на эмуляторе.

### Этап D: закрепление полноценной эксплуатации
- Сделать запуск воспроизводимым:
  - либо добавить `flutter`/`platform-tools` в user `PATH`;
  - либо оформить единый скрипт инициализации dev-сессии (PowerShell).
- Обновить документацию с финальным рабочим путем:
  - `docs/setup-summary.md` (последний статус после успешного `flutter run`);
  - `README.md` (короткий known-good раздел + действия при сетевых проблемах Chaquopy).

Критерий выхода этапа: новый запуск на этой машине проходит по инструкции без ручного расследования.

## Definition of Done (финальная готовность)
- `flutter run` стабильно собирает и устанавливает приложение на эмулятор.
