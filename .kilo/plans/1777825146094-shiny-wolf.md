# План внедрения алгоритма v2 (paper-masked) в приложение SeeDDetect

## Цель
Заменить текущую сегментацию для «сканов» (логика в `segment_seeds_scan.py`) на алгоритм из `AlgorithmVersions/VersionWithoutScan/run_pipeline_v2.py`, начиная **после получения изображения**, при этом:

- не переносить/не использовать GUI и CLI из `run_pipeline_v2.py`;
- сохранить текущий формат результата `analyze_image(...)` и набор артефактов (`all_mask`, `all_overlay`, `black_mask`, `black_overlay`) так, как это ожидает Flutter/Android слой;
- сохранить текущие коды ошибок и структуру `{"ok": ...}`.

## Контекст (что сейчас)
1. Мобильное приложение (Flutter) получает JSON из Android (Chaquopy) через метод `run_analysis_json`.
2. Android-код копирует входной файл во внутренний каталог и вызывает Python-функцию `segment_seeds_scan.analyze_image(input_path, output_dir, ...)`.
3. `analyze_image`:
   - читает изображение;
   - при необходимости даунскейлит (`_resize_if_needed` с `max_pixels/max_side`);
   - запускает `segment_all_seeds(prepared_img)` и `segment_black_seeds(prepared_img, all_mask)`;
   - пишет 4 артефакта (2 маски + 2 оверлея);
   - возвращает метрики и пути к артефактам.
4. Тест `tests/test_segment_contract.py` проверяет структуру payload и наличие этих 4 файлов.

## Интеграционная стратегия
Минимальная и безопасная для приложения замена: **обновить реализацию `analyze_image` и используемых в ней функций в `segment_seeds_scan.py`**, не меняя публичный контракт.

- Публичные точки входа остаются прежними:
  - `analyze_image(input_path, output_dir, ...) -> dict`
  - `process_image(...)` (адаптер)
  - `android_bridge.run_analysis_json(...)` не трогаем.
- Внутри `analyze_image` после чтения изображения внедряем pipeline v2:
  1. `detect_paper_mask` на подготовленном изображении.
  2. `crop_to_paper`.
  3. (Опционально) апскейл кропа до `target_short` (как в v2) для стабильности сегментации.
  4. `normalize_scan(..., paper_mask, do_shading)`.
  5. `white_balance_to_paper`.
  6. `segment_all_seeds` (версия v2, работающая в пределах paper-mask).
  7. `segment_black_seeds` (версия v2).
  8. `fixup_masks` (версия v2).
  9. Пересчёт контуров/счётчиков (как в v2) для финальных метрик.
  10. Запись артефактов в те же имена файлов, что сейчас.

## Что переносим из `run_pipeline_v2.py`
Переносим только вычислительную часть (без tkinter/argparse/батч-раннера):

1. Хелперы:
   - `odd`, `fill_holes`, `largest_component`, `component_bbox`
2. Детект бумаги и кроп:
   - `detect_paper_mask`, `crop_to_paper`
3. Нормализация/flat-field/WB:
   - `resize_short`
   - `estimate_paper_color`, `paper_pixel_mask`
   - `fit_poly_flatfield`, `estimate_background`
   - `paper_wb_and_levels`, `normalize_scan`, `white_balance_to_paper`
4. Сегментация:
   - `compute_delta_e`
   - `segment_all_seeds` (v2)
   - `segment_black_seeds` (v2)
   - `recover_black_pixels`, `interior_holes`, `fixup_masks`
5. Визуализация:
   - можно оставить текущие `visualize_all_seeds/visualize_black_seeds`, но адаптировать их к новым полям stats (например, `deltaE_thresh_used`, `L_core_used`).
   - либо минимально: переиспользовать `visualize_all`/`visualize_black` из v2, но обеспечить, чтобы результатом всё равно были ровно 2 jpg-оверлея.

Рекомендуемый вариант: **использовать визуализацию из v2** (`overlay_mask`, `put_text_block`, `visualize_all`, `visualize_black`) и удалить/не использовать нумерацию контуров из старой версии (она не нужна для контракта).

## Совместимость контракта результата
Оставляем без изменений:

- `ok`, `image`, `image_w`, `image_h`, `processing_ms`.
- `seed_count`, `all_seed_area_px`, `image_area_px`.
- `black_seed_count`, `black_seed_area_px`.
- `all_seed_ratio_pct`, `black_seed_ratio_pct`, `black_to_all_seed_ratio_pct`.
- `preprocessing` (результат `_resize_if_needed`).
- `artifacts`: ключи и смысл (4 пути) остаются прежними.

Изменение семантики допустимо только внутренне:

- `image_area_px` и ratio-проценты будут считаться по **обработанному кадру** (скорее всего кроп по бумаге), что логичнее для фото и не ломает UI/тесты.

Новые поля из v2 (`paper_detect`, `crop`, `audit`, и т.п.) по умолчанию **не добавляем**, чтобы не раздувать payload.

## Параметры и дефолты
1. Сохраняем существующие `max_pixels=12_000_000`, `max_side=4096`.
2. Добавляем в `analyze_image` необязательные параметры (с дефолтами), но не требуем их от Android:
   - `do_shading: bool = True` (аналог `--no-shading` в v2)
   - `target_short: int = 2400`

Если добавление новых аргументов нежелательно, можно захардкодить `do_shading=True`, `target_short=2400` внутри `analyze_image`.

## Ошибки/устойчивость
1. Сохраняем текущие error-коды `file_not_found`, `invalid_input`, `unsupported_format`, `decode_error`, `opencv_error`, `write_error`, `unexpected_error`.
2. Добавляем защитные fallback-и из v2:
   - если `detect_paper_mask` не нашёл бумагу: использовать full-frame mask.
3. Следим за Unicode-путями: оставляем `_safe_cv2_read/_safe_cv2_write` (использует `np.fromfile`/`tofile`).
