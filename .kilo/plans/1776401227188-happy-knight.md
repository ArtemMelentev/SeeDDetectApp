# План: поддержка PDF с извлечением изображения и русификация интерфейса

## Цель
Добавить в приложение два изменения:
1) поддержка выбора PDF-документа, извлечение из него изображения и передача извлеченного файла в Python-анализатор;
2) перевод пользовательского интерфейса на русский язык (без изменения названия приложения `Seed Detect`).

## Текущее состояние (по коду)
- Flutter UI (`lib/main.dart`) принимает только растровые форматы (`jpg/jpeg/png/tif/tiff/bmp`) и содержит англоязычные подписи кнопок, статусов и метрик.
- Android bridge (`android/app/src/main/kotlin/com/seeddetect/app/MainActivity.kt`) копирует входной файл как есть в `runDir` и передает его в Python.
- Python (`segment_seeds_scan.py`) валидирует расширение только как изображение и не умеет работать напрямую с PDF.
- Тексты ошибок, которые попадают в UI, сейчас в основном англоязычные.

## Принятая техническая стратегия
- Для PDF использовать нативное извлечение embedded-изображений на Android через `pdfbox-android` (а не рендер страницы в bitmap).
- Бизнес-правило: во всем PDF должно быть ровно одно извлекаемое изображение.
  - 0 изображений -> понятная ошибка для пользователя.
  - >1 изображения -> понятная ошибка для пользователя (согласно требованию).
  - 1 изображение -> сохраняем во временный файл в `runDir` и передаем путь в Python как обычное изображение.
- Для русификации на текущем этапе использовать перевод всех user-facing строк в `main.dart` + локализованные сообщения по `error.code`.

## План реализации

### Этап 1. Добавить обработку PDF в Android bridge
Файлы: `android/app/build.gradle`, `android/app/src/main/kotlin/com/seeddetect/app/MainActivity.kt`

1. Подключить зависимость для чтения PDF (например, `com.tom-roush:pdfbox-android`).
2. В native-пайплайне перед запуском Python определить тип входа:
   - если вход не PDF -> текущий путь без изменений;
   - если PDF -> ветка извлечения изображения.
3. Реализовать функцию извлечения единственного изображения из PDF:
   - открыть PDF через `resolveInputStream(...)`;
   - пройти по страницам и ресурсам (включая вложенные form XObject), собрать все `PDImageXObject`;
   - валидировать количество найденных изображений.
4. Ввести новые коды ошибок native-слоя:
   - `pdf_image_not_found`
   - `pdf_multiple_images`
   - `pdf_extract_failed`
5. При успешном извлечении сохранить изображение в `runDir` (PNG/JPG в зависимости от доступного формата) и передавать уже этот путь в `run_analysis_json(...)`.
6. Сохранить обратную совместимость текущего контракта payload (`ok/error`, существующие поля не ломать).

Ожидаемый результат: Python-скрипт по-прежнему получает путь к изображению, но вход может быть как image-файл, так и PDF с одним изображением.

Ожидаемый результат: пользователь может выбрать PDF тем же способом, что и изображение, и получает понятное объяснение при нарушении ограничения по количеству изображений.

## Критерии приемки (Definition of Done)
- Пользователь может выбрать `.pdf` через текущий file picker.
- Если в PDF больше одного изображения, в UI показывается понятное русское сообщение без падения.
- Если в PDF ровно одно изображение, оно извлекается и успешно анализируется существующим Python-скриптом.
- Кнопки, статусы, подписи метрик и подсказки в приложении отображаются на русском языке.
- Имя приложения `Seed Detect` не изменено.
- `flutter test` и `py -m unittest -v` проходят после изменений.

## Риски и меры
- PDF может содержать не raster image, а вектор/скан в сложной структуре -> возвращать `pdf_image_not_found` с понятным текстом.
- В больших PDF возможны пиковые затраты памяти -> обрабатывать потоково, не хранить лишние копии битмапов.
- Изображения могут быть вложены во form XObject -> рекурсивный обход ресурсов обязателен, иначе будут ложные ошибки «не найдено изображений».
