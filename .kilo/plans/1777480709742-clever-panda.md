# План: починить debug emulator и build release APK задачи

## Цель
- Починить VS Code задачи, чтобы стабильно работали два сценария:
  1) запуск эмулятора + debug (`Run SeedDetect debug (emulator)`),
  2) сборка релизного APK для Android (`Build SeedDetect release APK (phone)`).

## Что уже найдено
- В `.vscode/tasks.json` используются PowerShell-скрипты, но для задач не зафиксирован shell. Если в VS Code выбран `cmd`/Git Bash, команды с `$env:...` и `$flutter=...` ломаются.
- В задачах и `launch.json` зашиты абсолютные пути под конкретного пользователя (`C:\Users\Artem Melentyev\...`), что делает конфиг хрупким.
- В `android/settings.gradle` уже есть стандартный источник Flutter SDK через `android/local.properties` (`flutter.sdk`), значит путь можно резолвить динамически.
- Для релизной сборки в `android/app/build.gradle` есть строгая проверка signing-конфига (`android/key.properties` + keystore), поэтому задача сборки должна корректно прокидывать понятные ошибки, а не падать неявно.

## План изменений

### 1) Пересобрать `Start SeedDetect emulator` в устойчивый PowerShell-task
Файл: `.vscode/tasks.json`

- Явно указать shell для задач: `powershell.exe -NoProfile -ExecutionPolicy Bypass -Command` (на уровне задачи или общих `options`).
- Убрать жестко зашитые пути к `adb`/`emulator`:
  - сначала пробовать из `ANDROID_SDK_ROOT`/`ANDROID_HOME`,
  - затем fallback через `Get-Command adb(.exe)` / `Get-Command emulator(.exe)`.
- Проверять доступность AVD `SeedDetect_API34` (или fallback на первый доступный AVD), с понятной ошибкой если AVD не найден.
- Если нужный эмулятор уже запущен — переиспользовать его; если нет — стартовать.
- Ждать boot completion (`sys.boot_completed=1`) и разблокировать экран.

Результат: задача запуска эмулятора работает независимо от shell-профиля и не привязана к конкретному имени пользователя в пути.

### 2) Починить `Run SeedDetect debug (emulator)`
Файл: `.vscode/tasks.json`

- Оставить `dependsOn: Start SeedDetect emulator`.
- Динамически резолвить Flutter:
  - сначала из `android/local.properties` (`flutter.sdk`),
  - затем fallback `Get-Command flutter(.bat|.exe)`.
- Проставлять `ANDROID_HOME`/`ANDROID_SDK_ROOT` только если они реально обнаружены.
- Запускать `flutter run -d <целевой_emulator_serial> --device-timeout 120` через найденный бинарник.
- Добавить fail-fast проверки с читаемыми сообщениями (нет flutter/adb/sdk).

Результат: debug-задача стабильно стартует приложение на эмуляторе в один запуск.

### 3) Починить `Build SeedDetect release APK (phone)`
Файл: `.vscode/tasks.json`

- Аналогично резолвить Flutter и SDK без абсолютных путей.
- Оставить последовательность `flutter pub get` -> `flutter build apk --release --target-platform android-arm64`.
- `flutter clean` сделать опциональным (по умолчанию убрать, чтобы не замедлять каждую сборку; при необходимости добавить отдельную clean-задачу).
- Перед сборкой добавить явную проверку наличия `android/key.properties` и вывод понятной инструкции при отсутствии.

Результат: релизная сборка запускается предсказуемо и дает понятную диагностику по signing prerequisites.

### 4) Синхронизировать `launch.json` с задачами
Файл: `.vscode/launch.json`

- Для `SeedDetect (One Click)` убрать захардкоженный flutter path и перевести на запуск через починенную задачу (или полностью повторить ту же логику резолва, если это требуется конфигом).
- Для `SeedDetect (Dart Extension)` оставить `preLaunchTask: Start SeedDetect emulator`, без зависимостей от пользовательских абсолютных путей в launch-команде.

Результат: кнопка Debug в VS Code не ломается из-за старого пути SDK.

### 5) (Опционально) Привести `.vscode/settings.json` к переносимому виду
Файл: `.vscode/settings.json`

- Проверить необходимость `dart.flutterSdkPath` с абсолютным путем.
- Если мешает переносимости, убрать его или заменить на более переносимую схему (только если точно поддерживается расширением).

Результат: конфигурация workspace меньше зависит от конкретного ПК.

## Проверка после внедрения
- Запуск `Start SeedDetect emulator`:
  - эмулятор поднимается (или переиспользуется),
  - задача завершается без ошибок boot timeout.
- Запуск `Run SeedDetect debug (emulator)`:
  - flutter-приложение стартует на эмуляторе,
  - виден обычный debug-лог `flutter run`.
- Запуск `Build SeedDetect release APK (phone)`:
  - при корректном signing создается `build/app/outputs/flutter-apk/app-release.apk`,
  - при отсутствии signing — понятная ошибка с указанием, что нужно создать `android/key.properties`.

## Риски и как закрываем
- Риск: в системе несколько устройств (`adb`) и выбор не того target.
  - Мера: в задаче явно выбирать emulator serial (связанный с нужным AVD), не использовать "первое попавшееся устройство".
- Риск: shell в VS Code отличается (cmd/Git Bash/PowerShell Core).
  - Мера: принудительно задаем shell `powershell.exe` внутри самих задач.
- Риск: `local.properties` отсутствует на чистом клоне.
  - Мера: fallback на `Get-Command flutter`; при провале — явная инструкция по `flutter.sdk`.

## Объем изменений
- Основные правки: `.vscode/tasks.json`, `.vscode/launch.json`.
- Опциональная правка: `.vscode/settings.json` (только если подтверждена польза для переносимости).
